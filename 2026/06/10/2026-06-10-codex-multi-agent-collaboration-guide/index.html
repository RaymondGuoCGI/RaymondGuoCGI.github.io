实事求是：Codex 多 Agent 协作模式开启与使用的真实方案

实事求是：Codex 多 Agent 协作模式开启与使用的真实方案

如果你在网上看到一些教程说要到“Experimental”或“Features”标签页去开启多 Agent 模式，但在你的 Codex（v26.x 版本）设置界面里怎么也找不到，别怀疑，你没找错，是教程过时了。

经过对系统底层配置和功能列表的实测，以下是关于 Codex 多 Agent 协作模式的真实情况和使用方法。

1. 为什么你找不到“开启”开关？

事实： 在 Codex v26.x 及更高版本中，multi_agent 功能已经从“实验性”转为“稳定 (Stable)”，并且在系统层面是默认开启的。

你可以通过终端验证这一点：
打开终端，运行 codex features list，你会看到 multi_agent 这一项的状态是 true。这意味着你不需要去设置里找开关，它已经就在那里了。

2. 如何在界面中触发多 Agent 协作？

既然开关已经默认开启，协作模式并不是通过“勾选”来使用的，而是通过特定的交互动作来激活：

方法 A：使用 Composer (Ctrl + I / Cmd + I) —— 最核心的入口

多 Agent 协作的主要战场在 Composer。

  1. 在任何界面按下 Ctrl + I 打开 Composer 模式。
  2. 输入复杂的指令（例如：“重构整个项目的登录逻辑，并同步更新 API 文档和测试用例”）。
  3. 你会发现 Codex 会自动将任务拆分为多个子任务（Sub-tasks），并同时启动多个 Agent 并行处理。

方法 B：通过 @ 符号手动委派

在普通的聊天输入框中，你不再是面对一个单一的 AI：

  1. 输入 @ 符号。
  2. 你会看到一个智能体列表（如 @explore、@scout、@code 等）。
  3. 选择特定的 Agent，Codex 就会以协作模式启动一个专门的子线程来处理该领域的任务。

3. 如何监控协作进度？

当多个 Agent 同时工作时，你可以在左侧边栏的 工作树 (Worktrees) 中看到它们。

  • 每一个并行的 Agent 任务通常会关联一个独立的工作树，以确保它们在修改文件时不会互相冲突。
  • 你可以在 工作树 界面中看到哪些文件正在被哪个 Agent 修改，以及它们的审核进度。

4. 总结与建议

  • 不要再找开关了：如果你使用的是最新版本，它已经合入核心功能，不再提供显式的开关。
  • 认准入口：多 Agent 协作主要发生在 Composer 模式下。
  • 查看状态：利用侧边栏的 “工作树” 和 “工作树”图标 来管理并行的 AI 任务。

如果你在设置中确实需要配置，请查看 “配置” (Config) 页面（那个带点的齿轮图标），那里可以调整模型偏好（如推理强度），但“开启”本身已经不需要你操心了。
