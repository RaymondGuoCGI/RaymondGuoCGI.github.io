从一个想法到上线：我把单图转 PBR 材质工具做成了 pbr.raymondstudio.cn

前几天我把一个一直想做的小工具真正落地了：

上传一张普通材质参考图，自动生成一组可用于渲染的 PBR 贴图。

现在它已经可以直接访问：

  • pbr.raymondstudio.cn

这篇文章不打算写成“发布公告”，而是把这个项目从想法、功能定义、界面迭代、导出逻辑、域名上线，到广告准备的完整过程记录下来。

如果你也在做独立工具站、3D 工作流工具，或者只是想把一个具体需求快速做成产品，这个过程应该会有参考价值。

一、最初的需求其实非常具体

这个工具的出发点不是“我想做一个 AI 网站”，而是一个非常实际的 3D 工作流问题：

  • 手上只有一张普通的水泥墙、木纹、布料、石材图片
  • 没有现成的扫描级 PBR 材质
  • 但希望能尽快拿去做 lookdev、概念验证或者快速渲染测试

所以目标很明确：

  1. 用户上传一张图片
  2. 浏览器本地分析并生成常见 PBR 通道
  3. 支持不同分辨率
  4. 支持不同渲染器导出命名
  5. 最终一键打包下载

换句话说，它不是一个“贴图展示站”，而是一个真正解决轻量材质整理问题的工具。

二、第一版能跑，但离“可用产品”差得很远

一开始实现核心逻辑其实并不难：

  • 读入图片
  • 生成 Base Color / Height / Normal / Roughness / AO / Metallic
  • 提供单通道下载

但只做到这里，离用户真正愿意用还差很多。

很快就暴露出几个典型问题：

  • 页面排版太竖，必须一直往下滚
  • 六个通道预览的尺寸和对齐不一致
  • “下载全部”不是 ZIP
  • 通道名称没有按渲染器工作流区分
  • 保持原图比例只作用于预览，没有作用于输出
  • 浅色 / 深色模式的视觉适配不完整

这类问题看似都不是“核心算法”，但它们恰恰决定了一个工具是“能跑”还是“能用”。

三、我后来发现，真正花时间的其实不是生成，而是产品化

这个项目后续的大部分时间都花在“把一个功能做成产品”上，而不是继续堆算法。

1. 横向布局重构

我最后把主界面改成了三栏结构：

  • 左侧：源图预览
  • 中间：输出通道
  • 右侧：生成设置

这样用户不需要一直向下滚动，就能同时看到：

  • 输入图
  • 六个输出通道
  • 参数设置

这个改动非常关键，因为它直接影响“第一次打开网站时是否觉得顺手”。

2. 上传区域和预览区域合并

后面我把原本独立的上传模块删掉了，改成：

  • 直接把图片拖进“输入纹理”预览框
  • 预览框本身同时承担点击上传与拖拽上传

这一步让交互更直接，也让页面少了一块重复区域。

3. 通道输出模块反复微调

六张通道图这个模块，其实来回改了很多次，包括：

  • 两排三列严格对齐
  • 每张卡片高度统一
  • 标题区和下载按钮不重叠
  • 小屏时不挤压
  • 下载按钮改成图标式，减少横向占用

这些都属于“用户一眼就会感受到顺不顺”的地方。

四、导出目标不是简单改名字，背后其实是工作流映射

这个项目里一个比较重要的迭代，是把“导出目标”真正做成工作流分类，而不是纯 UI 选项。

现在支持的导出目标包括：

  • Generic Metal/Rough
  • Blender
  • Unreal Engine
  • V-Ray
  • KeyShot

这里最开始我也踩了坑。

比如一开始给 KeyShot 的通道里放了：

  • Displacement
  • Bump

这两个在很多情况下对用户来说是重复表达，反而不够实用。后面重新梳理后，才改成更贴近日常使用的方向，例如：

  • Base Color
  • Roughness
  • Bump / Normal
  • Specular / Reflection Color
  • Opacity（按目标工作流决定）

V-Ray 和 KeyShot 这类渲染器，其实并不能简单照搬标准金属度工作流的命名。所以后来我补了：

  • specular
  • reflection color
  • opacity

这类衍生输出，让导出结果不只是“六张图”，而是更接近真正能塞进不同渲染器材质节点里的资源整理结果。

五、保持原图尺寸与无缝处理，是两个真正提高可用性的选项

后面我又加了两个开关：

保持原图尺寸

这个选项的作用是：

  • 勾选后：按原图宽高比输出六张通道图
  • 不勾选：统一裁成正方形输出

这个改动最开始其实只影响了预览，后来发现真正重要的是：

输出通道本身也必须和原图保持一致的比例

否则用户会看到预览正确，但下载结果还是正方形，这会直接破坏信任感。

无缝

这个选项用于对普通图片做基础边缘融合，尽量减轻平铺时的接缝问题。

因为真实情况是：

  • 大多数网上找来的图片不是扫描材质
  • 没有做过无缝处理
  • 直接平铺会非常明显地露馅

所以哪怕只是基础版的 seam reduction，也比没有要强很多。

六、看起来很小的参数设计，实际上也会影响用户理解

例如后来我把：

  • 法线强度
  • 高度细节

从原本偏工程味的范围，改成了：

  • 最左 -1
  • 中间 0
  • 最右 1

也就是一个“减弱 / 中性 / 增强”的对称控制方式。

这件事的意义在于：

  • 用户更容易理解
  • 中间值就是默认状态
  • 不需要知道 4 和 3 这种上限到底是什么意思

同时我保留了内部映射，不会因为负值直接把法线反转成奇怪结果。

这种看似只是滑块 UI 的改动，本质上是在降低理解成本。

七、域名、部署和真正上线，比写完代码更像“临门一脚”

项目真正从本地工具变成在线工具，是在接上域名以后。

最终使用的是：

  • 部署平台：Vercel
  • 自定义域名：pbr.raymondstudio.cn

这个过程里也有几个典型问题：

  • 子域名 DNS 指向
  • Vercel 对自定义域名的解析要求
  • Cloudflare 中记录是否正确
  • 为什么根域或其他子域可用，这个子域却还没生效

这些问题如果没踩过，会觉得“明明记录已经加了，为什么还是不通”。

最后的经验很简单：

  • DNS 配置要和托管平台要求严格一致
  • 改完之后要给解析生效时间
  • 看状态不要只看 Cloudflare，还要看 Vercel 后台是否已经识别成功

当 pbr.raymondstudio.cn 真正能打开的时候，这个项目才算完成了第一阶段闭环。

八、上线以后，我又补了两件很现实的东西：反馈和广告准备

1. 反馈入口

主流网站一般都会有一套用户反馈入口。

所以后来我给这个工具站加了：

  • 右下角反馈按钮
  • 反馈弹窗
  • 反馈类型选择
  • 问题描述
  • 邮箱（可选）
  • 自动附带当前参数上下文

当前这版为了轻量，采用的是邮件方式，不需要后端就能先收集反馈。

2. AdSense 准备

既然是工具站，后面自然也会考虑广告变现。

所以我顺手把这些基础设施也补了：

  • About
  • Privacy
  • Terms
  • FAQ
  • robots.txt
  • sitemap.xml
  • ads.txt
  • AdSense 脚本预埋

这里也有一个现实问题：

子域工具站是否能通过 AdSense，不只取决于它自己，还可能受根域内容结构影响。

这也是为什么后来我又回过头去整理根域博客内容，让整个域名体系更干净。

九、这个项目让我再次确认：独立工具最重要的是“明确解决一个具体问题”

如果只从技术上看，这个网站并不复杂：

  • 前端页面
  • 浏览器侧图像处理
  • 通道推断
  • ZIP 导出

真正让它有意义的，是它解决的是一个非常具体的问题：

  • 我有一张图
  • 我没有完整 PBR 材质
  • 我现在就想拿去渲染测试

工具站最怕的是做成“看起来功能很多，但其实没有明确使用场景”。

而这个工具的好处在于，它的目标始终很集中：

  • 快速
  • 本地处理
  • 明确导出
  • 面向真实 3D 使用场景

十、后续还想继续补什么

这个项目现在已经能用了，但还远没有结束。

后面我比较想继续补的方向包括：

  • 更好的无缝处理
  • 更精细的材质类型判断
  • 更准确的渲染器导出预设
  • 更多格式支持
  • 更完善的反馈与问题追踪

如果后续我继续迭代它，应该还会再写一篇“第二阶段复盘”。

结语

对我来说，这个项目最有意思的地方不只是“做出了一个单图转 PBR 的网站”，而是把一个非常具体的工作流需求，真正从想法推进到了：

  • 可访问
  • 可使用
  • 可导出
  • 可收集反馈
  • 可继续迭代

这比单纯写一个 demo 更接近真正的产品。

如果你也在做类似的小工具，我越来越觉得一件事很重要：

先不要追求功能大全，先把一个具体问题解决到足够顺手。

这通常比“做一个看起来什么都能做的平台”更有价值。
